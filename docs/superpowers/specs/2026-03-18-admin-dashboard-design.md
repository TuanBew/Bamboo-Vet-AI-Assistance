## 1. Overview

Bamboo Vet is a single Next.js monorepo (currently v16, described throughout as "Next.js 15 App Router" тАФ App Router pattern is unchanged) containing two products:

- **Product A** тАФ Customer-facing bilingual AI veterinary chatbot (`/`, `/chat`, `/app`). Already built. No changes.
- **Product B** тАФ Internal admin SaaS dashboard (`/admin/*`). This spec covers Product B entirely.

The admin dashboard is a Vietnamese-only analytics and management platform visible exclusively to users with `is_admin = true`. It provides platform usage analytics, user/clinic management, knowledge base stats, and a pivot-table-heavy data explorer for the veterinary chatbot platform.

## 3. Database Architecture

### 3.1 New Tables

#### `profiles`
Extends `auth.users`. Auto-populated via `SECURITY DEFINER` trigger on user creation.

```sql
CREATE TABLE profiles (
  id              uuid PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  full_name       text,
  email           text,
  is_admin        boolean NOT NULL DEFAULT false,
  clinic_name     text,
  clinic_type     text,         -- 'phong_kham' | 'nha_thuoc' | 'thu_y' | 'my_pham' | 'cho' | 'khac'
  facility_code   text UNIQUE,  -- M├г c╞б sс╗Я (unique per clinic, groups staff under one clinic)
  staff_code      text,         -- M├г nh├вn vi├кn (individual staff identifier within a clinic)
  address         text,
  street          text,
  ward            text,         -- Ph╞░с╗Эng/X├г
  district        text,         -- Quс║нn/Huyс╗Зn
  province        text,         -- Tс╗Йnh/Th├аnh phс╗С
  region          text,         -- 'Bс║пc' | 'Trung' | 'Nam'
  zone            text,         -- V├╣ng kinh tс║┐
  latitude        numeric(9,6),
  longitude       numeric(9,6),
  avatar_url      text,
  created_at      timestamptz NOT NULL DEFAULT now()
);
```

**Trigger (migration: `YYYYMMDD_add_profile_trigger.sql`):**
Must use `SECURITY DEFINER` so the function runs as the owner (bypassing RLS on insert).

```sql
CREATE OR REPLACE FUNCTION handle_new_user()
RETURNS trigger AS $$
BEGIN
  INSERT INTO profiles (id, email)
  VALUES (NEW.id, NEW.email);
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION handle_new_user();
```

**Admin promotion:** Done manually via Supabase SQL editor:
```sql
UPDATE profiles SET is_admin = true WHERE email = 'admin@example.com';
```

**RLS:**
- Users can `SELECT` and `UPDATE` their own row (`id = auth.uid()`)
- Service role can `SELECT` all rows (for admin API)
- No direct client-side `INSERT` (trigger handles it; trigger function is `SECURITY DEFINER`)

#### `chat_analytics`
Per-conversation metadata for category analytics. Not auto-generated by the chatbot тАФ populated entirely by the seed script.

```sql
CREATE TABLE chat_analytics (
  id                uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  conversation_id   uuid NOT NULL REFERENCES conversations(id) ON DELETE CASCADE,
  user_id           uuid NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  drug_group        text,   -- 'kh├бng sinh' | 'vitamin' | 'hormone' | 'vс║пc-xin' | 'kh├бng k├╜ sinh tr├╣ng' | 'kh├бc'
  animal_type       text,   -- 'tr├вu b├▓' | 'lс╗гn' | 'g├а' | 'ch├│ m├иo' | 'thс╗зy sс║гn' | 'kh├бc'
  query_type        text,   -- 'chс║йn ─Сo├бn' | '─Сiс╗Бu trс╗Л' | 'ph├▓ng bс╗Зnh' | 'liс╗Бu l╞░с╗гng' | 'kh├бc'
  created_at        timestamptz NOT NULL DEFAULT now()
);
```

**RLS:** Service role only. No client-side access.

**Note:** Because `chat_analytics` rows are seeded (not auto-generated), only seeded conversations will appear in analytics views. Real user conversations from Product A (before seeding) will not appear in analytics тАФ this is by design for the internal tool.

### 3.2 Materialized Views

All heavy aggregations are pre-computed in PostgreSQL. API routes execute plain `SELECT` queries тАФ no per-request computation. Views are refreshed via `REFRESH MATERIALIZED VIEW CONCURRENTLY`.

#### `mv_monthly_queries`
Drives monthly time-series charts, sparklines, user pivot tables.

```sql
CREATE MATERIALIZED VIEW mv_monthly_queries AS
SELECT
  ca.user_id,
  EXTRACT(YEAR  FROM c.created_at)::int AS year,
  EXTRACT(MONTH FROM c.created_at)::int AS month,
  COUNT(DISTINCT c.id)                   AS session_count,
  COUNT(m.id)                            AS query_count
FROM conversations c
JOIN messages m      ON m.conversation_id = c.id AND m.role = 'user'
JOIN chat_analytics ca ON ca.conversation_id = c.id
GROUP BY ca.user_id, year, month;

CREATE UNIQUE INDEX ON mv_monthly_queries (user_id, year, month);
```

#### `mv_daily_queries`
Drives the clinic detail modal (day-by-day breakdown). Filtered by `user_id`.

```sql
CREATE MATERIALIZED VIEW mv_daily_queries AS
SELECT
  ca.user_id,
  EXTRACT(YEAR  FROM c.created_at)::int AS year,
  EXTRACT(MONTH FROM c.created_at)::int AS month,
  EXTRACT(DAY   FROM c.created_at)::int AS day,
  COUNT(DISTINCT c.id)                   AS session_count,
  COUNT(m.id)                            AS query_count
FROM conversations c
JOIN messages m       ON m.conversation_id = c.id AND m.role = 'user'
JOIN chat_analytics ca ON ca.conversation_id = c.id
GROUP BY ca.user_id, year, month, day;

CREATE UNIQUE INDEX ON mv_daily_queries (user_id, year, month, day);
```

#### `mv_category_stats`
Drives donut/pie charts. Includes `province` and `clinic_type` so dashboard filters can be applied.

```sql
CREATE MATERIALIZED VIEW mv_category_stats AS
SELECT
  EXTRACT(YEAR  FROM c.created_at)::int AS year,
  EXTRACT(MONTH FROM c.created_at)::int AS month,
  p.province,
  p.clinic_type,
  ca.drug_group,
  ca.animal_type,
  ca.query_type,
  COUNT(*)                               AS count
FROM conversations c
JOIN chat_analytics ca ON ca.conversation_id = c.id
JOIN profiles p        ON p.id = ca.user_id
GROUP BY year, month, p.province, p.clinic_type,
         ca.drug_group, ca.animal_type, ca.query_type;

-- Required for REFRESH CONCURRENTLY
CREATE UNIQUE INDEX ON mv_category_stats (year, month, province, clinic_type, drug_group, animal_type, query_type);
```

#### `mv_dashboard_kpis`
Single-row aggregate snapshot for KPI cards. Uses LEFT JOINs so all conversations are counted regardless of `chat_analytics` presence.

**Note:** This view produces a single aggregate row with no natural unique key, so `REFRESH MATERIALIZED VIEW CONCURRENTLY` is **not possible** for this view. The `refresh-views.ts` script must use plain `REFRESH MATERIALIZED VIEW mv_dashboard_kpis` (without `CONCURRENTLY`) for this view only. All other 3 views use `CONCURRENTLY`.

```sql
CREATE MATERIALIZED VIEW mv_dashboard_kpis AS
SELECT
  COUNT(DISTINCT c.id)                                       AS total_sessions,
  COUNT(m.id) FILTER (WHERE m.role = 'user')                AS total_queries,
  COUNT(DISTINCT c.user_id)                                  AS total_users,
  (SELECT COUNT(*) FROM kb_documents WHERE status = 'active') AS total_documents,
  (SELECT COUNT(*) FROM profiles WHERE is_admin = false)     AS total_staff,
  now()                                                      AS refreshed_at
FROM conversations c
LEFT JOIN messages m ON m.conversation_id = c.id;

-- No unique index тАФ this view cannot use REFRESH CONCURRENTLY (single aggregate row).
-- Use: REFRESH MATERIALIZED VIEW mv_dashboard_kpis;
```

**Refresh strategy:** Script `scripts/refresh-views.ts` refreshes all 4 views:
- `mv_monthly_queries`, `mv_daily_queries`, `mv_category_stats` тЖТ `REFRESH MATERIALIZED VIEW CONCURRENTLY`
- `mv_dashboard_kpis` тЖТ `REFRESH MATERIALIZED VIEW` (non-concurrent, ~instant for a single row)

Triggered manually via "L├аm mс╗Ыi dс╗п liс╗Зu" button in the admin top bar (internal server action, not a public API).

## 4. Authentication & Security

### 4.1 Critical Pre-Requisite: Fix Middleware Wiring

**`proxy.ts` is not currently active as Next.js middleware.** Next.js App Router only recognises a middleware file named `middleware.ts` (or `middleware.js`) at the project root. The existing `proxy.ts` exports a function named `proxy` тАФ Next.js ignores it entirely. No middleware runs today.

**Required fix (first task of implementation):**
1. Rename `proxy.ts` тЖТ `middleware.ts` at the project root
2. Rename the exported function from `proxy` to `middleware`
3. The `config` export (matcher) stays exactly as-is

After this rename, `lib/supabase/middleware.ts`'s `updateSession()` will be correctly invoked on every non-static request.

---

### 4.2 Admin Role Detection

`is_admin` lives in `profiles.is_admin`. Default `false`. Promoted manually via SQL.

**File to modify for admin guards: `lib/supabase/middleware.ts`** (the `updateSession()` function).

**Login redirect logic (extended `updateSession()`):**

After the existing session refresh, the middleware adds:
1. For `/admin/*` routes: fetch `profiles.is_admin` using **`createServiceClient()`** from `lib/supabase/server.ts` (service role, bypasses RLS тАФ the anon client cannot read other users' `profiles` rows). If no session or `is_admin = false` тЖТ redirect to `/login`. If `is_admin = true` тЖТ allow.
2. For `/app` and `/app/*` routes: after session is confirmed, check `profiles.is_admin`. If `true` тЖТ redirect to `/admin/dashboard`.

**The login flow for admins (two-step redirect):**
```
Admin logs in at /login
  тЖТ Supabase session created
  тЖТ Existing logic redirects to /app
  тЖТ Middleware intercepts /app request
  тЖТ createServiceClient() fetches profiles.is_admin = true
  тЖТ Redirects to /admin/dashboard
```

**Performance note:** The `is_admin` DB fetch adds ~10тАУ30ms per request on `/admin/*` and `/app` routes. Acceptable for an internal tool.

**Service role client:** Use `createServiceClient()` from `lib/supabase/server.ts` тАФ this is the existing service role client that bypasses RLS.

### 4.3 `requireAdmin()` Utility

Location: `lib/admin/auth.ts`

```typescript
import { NextResponse } from 'next/server'
import { createServiceClient } from '@/lib/supabase/server'

type AdminContext = { user: User; profile: Profile }
type AdminResult  = AdminContext | NextResponse

export async function requireAdmin(): Promise<AdminResult> {
  // 1. Get session via createServiceClient()
  // 2. If no session тЖТ return NextResponse.json({ error: 'Forbidden' }, { status: 403 })
  // 3. Fetch profiles row: SELECT * FROM profiles WHERE id = user.id
  // 4. If !profile || !profile.is_admin тЖТ return NextResponse.json({ error: 'Forbidden' }, { status: 403 })
  // 5. Return { user, profile }
}
```

**Usage pattern in every `/api/admin/*` route:**
```typescript
export async function GET(request: Request) {
  const auth = await requireAdmin()
  if (auth instanceof NextResponse) return auth   // early return on 403

  const { user, profile } = auth
  // proceed тАФ service role client for DB queries
  const db = createServiceClient()
  // ...
}
```

## 5. API Routes

All routes under `/api/admin/*`. All protected by `requireAdmin()`. All return JSON. All use the service role Supabase client.

### `GET /api/admin/dashboard`
Query params: `year`, `month`, `province`, `clinic_type`

**KPI filter behaviour:** The 5 KPI cards (`total_queries`, `total_sessions`, etc.) always show **platform-wide totals** тАФ they are not affected by `province`/`clinic_type` filters. Only the charts, category stats, user table, and clinic map respond to filters. This is intentional: the KPIs come from `mv_dashboard_kpis` (no filter dimensions) and serve as the global baseline. The filter bar label should note "Bс╗Щ lс╗Нc ├бp dс╗еng cho biс╗Гu ─Сс╗У" (filter applies to charts).

Response shape:
```typescript
{
  kpis: {
    total_queries: number;      // platform-wide, unfiltered
    total_sessions: number;     // platform-wide, unfiltered
    total_users: number;        // platform-wide, unfiltered
    total_documents: number;    // platform-wide, unfiltered
    total_staff: number;        // platform-wide, unfiltered
    refreshed_at: string;
  };
  monthly_series: Array<{
    year: number;
    month: number;
    query_count: number;
    session_count: number;
    is_forecast: boolean;    // true for the 3 projected months
  }>;
  category_stats: {           // filtered by province + clinic_type if provided
    drug_groups:  Array<{ name: string; count: number }>;
    animal_types: Array<{ name: string; count: number }>;
    query_types:  Array<{ name: string; count: number }>;
  };
  top_users: Array<{
    user_id: string;
    full_name: string;
    clinic_name: string;
    clinic_type: string;
    total_queries: number;
    total_sessions: number;
    monthly_sparkline: number[];
  }>;
  clinic_map: Array<{
    user_id: string;
    clinic_name: string;
    clinic_type: string;
    province: string;
    latitude: number;
    longitude: number;
    total_queries: number;
  }>;
  top_clinics: Array<{ clinic_name: string; total_queries: number }>;
}
```

**Forecast computation (API-side):**
The API route computes the 3-month forecast from `mv_monthly_queries` data:
1. Take the last 6 months of real `monthly_series` data (aggregated across all users matching filters)
2. Apply simple linear regression: `y = a + b*x` where `x` is month index (0тАУ5), `y` is `query_count`
3. Extrapolate for months `current+1`, `current+2`, `current+3`
4. Append these 3 rows to `monthly_series` with `is_forecast: true`

Client renders forecast points as a dotted line using Recharts' `strokeDasharray`.

### `GET /api/admin/new-activity`
Query params: `year`, `month`

Response shape:
```typescript
{
  kpis: {
    total_new_sessions: number;
    total_new_queries: number;
    total_new_users: number;
    avg_queries_per_session: number;          // total_new_queries / total_new_sessions
    total_new_documents: number;
    avg_session_duration_min: number;
    // avg_session_duration_min computation:
    // For each conversation in the month, compute duration as:
    //   EXTRACT(EPOCH FROM (MAX(m.created_at) - MIN(m.created_at))) / 60
    // Then average these durations across all conversations in the month.
    // Conversations with only 1 message count as 1 minute.
    // Computed via raw SQL in the API route (not a materialized view).
  };
  daily_query_volume: Array<{ day: number; query_count: number }>;
  daily_sessions:     Array<{ day: number; session_count: number }>;
  recent_sessions: Array<{
    session_code: string;    // first 8 chars of conversation.id uppercased, e.g. "A3F7C219"
    date: string;
    user_name: string;
    query_count: number;
    duration_min: number;   // computed as: EXTRACT(EPOCH FROM (MAX(m.created_at) - MIN(m.created_at)))/60
                            // for all messages in the conversation. Minimum 1 min.
  }>;
  top_questions: Array<{ question_prefix: string; count: number }>;
  // top_questions derivation: take all user messages in the selected month, truncate content
  // to first 60 characters, group by this prefix, count occurrences, return top 10 by count.
  // Computed from `messages` table directly (not a materialized view тАФ dataset is small per month).
  category_stats: {
    drug_groups:  Array<{ name: string; count: number }>;
    animal_types: Array<{ name: string; count: number }>;
```

## 12. Decisions Log

| Decision | Choice | Rationale |
|---|---|---|
| Admin role storage | `profiles.is_admin` | Standard Supabase pattern, RLS-compatible, auto-created by SECURITY DEFINER trigger |
| Data aggregation | PostgreSQL materialized views | Fast reads via plain SELECT; no per-request computation |
| Category filter on dashboard | `mv_category_stats` includes `province` + `clinic_type` columns | Enables server-side filtering without raw table joins on request |
| KPI counts | LEFT JOIN (all conversations) | INNER JOIN on `chat_analytics` would silently exclude real user conversations |
| Forecast algorithm | Server-side linear regression, last 6 months | Deterministic, simple, no ML dependency; result included in API response |
| AI/RAGflow in admin | None | Analytics-only tool; AI chat is Product A only |
| Admin language | Vietnamese only | Internal tool, all operators are Vietnamese |
| "Xem lс╗Лch sс╗н" endpoint | New `/api/admin/users/[userId]/conversations` routes | Existing route is user-scoped (RLS); admin needs service-role bypass |
| Clinic pivot row key | `facility_code` (UNIQUE) | Clinics group multiple staff; `user_id` would create duplicate clinic rows |
| Export libraries | `xlsx` + `jspdf` + `jspdf-autotable` | Standard, well-maintained, tree-shakeable |
| Middleware wiring | Rename `proxy.ts` тЖТ `middleware.ts`, rename export `proxy` тЖТ `middleware` | Next.js only runs root `middleware.ts`; `proxy.ts` is currently ignored entirely |
| Service role function | `createServiceClient()` from `lib/supabase/server.ts` | This is the actual function name in the codebase тАФ not `createServiceRoleClient()` |
| Middleware logic file | `lib/supabase/middleware.ts` (`updateSession`) | Actual guard logic goes here after proxyтЖТmiddleware rename |
| Admin login redirect | Two-hop: `/login` тЖТ `/app` тЖТ `/admin/dashboard` | Reuses existing post-login redirect; middleware intercepts `/app` for admins |
| KPI filter scope | Platform-wide totals only (unfiltered) | `mv_dashboard_kpis` has no filter dimensions; filter bar applies to charts only |
| Trigger SECURITY DEFINER | Required | RLS on `profiles` blocks trigger insert otherwise |
| `/admin/settings` | Minimal: profile display + refresh button | Internal tool needs no self-serve role management |
