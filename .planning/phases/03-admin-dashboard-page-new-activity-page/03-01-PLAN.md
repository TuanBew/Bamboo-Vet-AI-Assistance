**Exported function: `computeForecast(data: MonthlyDataPoint[], forecastMonths?: number): ForecastPoint[]`**

Algorithm:
1. Sort input by date ascending (year * 12 + month)
2. Take last 6 data points as `recent`
3. If `recent.length < 2`, return all points with `is_forecast: false` (not enough data)
4. Compute linear regression: `y = a + b*x` where x = index (0 to n-1), y = query_count
   - `b = (n * sumXY - sumX * sumY) / (n * sumX2 - sumX * sumX)`
   - `a = (sumY - b * sumX) / n`
5. For each forecast month (default 3):
   - Compute next year/month by incrementing from last real point (handle year rollover when month > 12)
   - Predict `query_count = Math.max(0, Math.round(a + b * (n - 1 + i)))` where i starts at 1
   - Predict `session_count = Math.round(predictedQueries * 0.25)` (rough session ratio)
   - Set `is_forecast: true`
6. Return `[...sortedRealPoints.map(d => ({...d, is_forecast: false})), ...forecastPoints]`

**Critical detail from RESEARCH.md:** The last real data point must be duplicated as the first forecast point for visual continuity in the chart. Actually, the bridge point is handled client-side by the chart component (it reads both real and forecast series). The forecast utility just returns clean data. The client component will handle the bridge by including the last real point in the forecast Line series.

No `'use client'` directive -- this runs server-side only.

**Exported function: `getDashboardData(filters: DashboardFilters): Promise<DashboardData>`**

Implementation steps:
1. `const db = createServiceClient()`
2. **KPIs (unfiltered, always platform-wide):** `SELECT * FROM mv_dashboard_kpis LIMIT 1` -- parse numeric fields with `Number()` (Supabase may return strings for bigint).
3. **Monthly series:** Query `mv_monthly_queries` aggregated across all users: `SELECT year, month, SUM(query_count) as query_count, SUM(session_count) as session_count FROM mv_monthly_queries GROUP BY year, month ORDER BY year, month`. Apply province/clinic_type filter by joining profiles if filters provided. Pass result to `computeForecast()`.
4. **Category stats:** Query `mv_category_stats` with year/month from `filters.month` (parse year and month from "2026-03"). Apply province/clinic_type filter if provided. Aggregate by drug_group, animal_type, query_type separately. Return 3 arrays sorted by count desc.
5. **Daily volume:** Query `mv_daily_queries` for the selected month: `SELECT day, SUM(query_count)::int as query_count FROM mv_daily_queries WHERE year = $year AND month = $month GROUP BY day ORDER BY day`. Parse year and month from `filters.month`. Parse numeric results with `Number()`.
6. **Top users (top 20):** Query `mv_monthly_queries` joined with `profiles` to get user details. Aggregate total_queries and total_sessions per user. Sort by total_queries desc, limit 20. For each user, compute:
   - `monthly_sparkline`: array of 12 numbers (last 12 months query_count) from mv_monthly_queries
   - `avg_queries`: total_queries / number of active months
   - `days_active`: count of distinct (year, month, day) from mv_daily_queries for that user
   - `drug_group_breakdown`: from mv_category_stats grouped by drug_group for that user's conversations
   - `query_type_breakdown`: from mv_category_stats grouped by query_type for that user's conversations
7. **Clinic map:** Query profiles where latitude IS NOT NULL AND longitude IS NOT NULL AND is_admin = false. Join with aggregated query counts from mv_monthly_queries. Return all clinics with coordinates.
8. **Top clinics (top 10):** From clinic_map data, sort by total_queries desc, limit 10. Return clinic_name + total_queries.

**IMPORTANT:** All numeric values from Supabase must be parsed with `Number()` to avoid string type issues in charts (Pitfall 7 from RESEARCH.md).

```typescript
import { NextRequest, NextResponse } from 'next/server'
import { requireAdmin } from '@/lib/admin/auth'
import { getDashboardData } from '@/lib/admin/services/dashboard'

export async function GET(request: NextRequest) {
  const auth = await requireAdmin()
  if (auth instanceof NextResponse) return auth

  const { searchParams } = new URL(request.url)
  const month = searchParams.get('month') || new Date().toISOString().slice(0, 7)
  const province = searchParams.get('province') || ''
  const clinic_type = searchParams.get('clinic_type') || ''

  try {
    const data = await getDashboardData({ month, province, clinic_type })
    return NextResponse.json(data)
  } catch (error) {
    console.error('Dashboard API error:', error)
    return NextResponse.json({ error: 'Internal server error' }, { status: 500 })
  }
}
```

The route handler is thin -- it does auth guard, parses params, calls the service, returns JSON. The service function contains all query logic so the Server Component (page.tsx) can call it directly for SSR without going through HTTP.
